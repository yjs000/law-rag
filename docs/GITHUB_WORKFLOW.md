# GitHub 이슈와 PR 운영

상태: 사용 가능

최종 갱신: 2026-07-13

이 문서는 이슈, 프로젝트 보드, PR에서 사용하는 최소 공통 규칙을 정의한다. 제품·설계 결정은 각 권위 문서에 기록하고 이슈 본문에 복제하지 않는다.

## 이슈

이슈 하나는 검증 가능한 결과 하나를 다룬다. 문제, 기대 결과 또는 재현 방법, 완료 조건을 적고 관련 제품 명세나 설계 문서를 연결한다. 개인정보, 인증정보, 실제 사건자료와 법률 원문 전문은 첨부하지 않는다.

라벨은 다음 네 개만 기본으로 사용한다.

| 라벨 | 용도 |
|---|---|
| `type: bug` | 기대 동작과 다른 오류 또는 회귀 |
| `type: feature` | 사용자 기능, 품질 또는 내부 개선 |
| `type: docs` | 문서만 변경하는 작업 |
| `status: blocked` | 외부 결정·권한·데이터가 없으면 진행할 수 없는 작업 |

유형 라벨은 하나만 단다. `status: blocked`는 차단 사유와 해제 조건을 댓글이나 본문에 적은 경우에만 추가한다. 컴포넌트, 난이도, 담당자 이름은 기본 라벨로 만들지 않고 검색, 담당자, 프로젝트 필드로 관리한다.

## 프로젝트 보드

프로젝트 3은 실행 상태를 보여 주는 단일 보드로 사용한다.

- `Backlog`: 범위나 순서가 아직 확정되지 않은 후보 작업
- `Ready`: 완료 조건이 있고 바로 시작할 수 있는 작업
- `In progress`: 현재 실제로 작업 중인 항목
- `In review`: 구현을 마치고 리뷰 또는 최종 검증 중인 항목
- `Done`: 코드·문서·검증이 완료된 항목

우선순위가 필요하면 프로젝트의 단일 선택 필드 `Priority`만 사용한다.

- `P0`: 보안, 데이터 손실, 근거 없는 법률 답변처럼 즉시 중단·대응할 문제
- `P1`: 현재 목표나 배포를 막는 문제
- `P2`: 나머지 계획 작업. 기본값이며 명시하지 않아도 된다.

동시에 진행 중인 항목 수를 작게 유지한다. 상태 변경은 실제 작업 상태와 함께 갱신하고, 완료되지 않은 후속 작업은 `Done` 항목에 숨기지 말고 새 이슈로 분리한다.

## PR

PR은 가능하면 이슈를 `Closes #번호`로 연결하고 한 가지 결과만 포함한다. 본문에는 변경 이유, 핵심 변경, 실행한 검증 명령과 결과, 문서·스키마·배포 영향을 적는다.

검색·답변 변경은 고정 평가셋 회귀와 인용 검증을 포함한다. 데이터 변경은 출처, 원문 해시, 수집·파서 버전의 추적성이 유지되는지 확인한다. 중요한 결정은 관련 설계 문서의 결정 기록에 날짜와 이유를 남긴다.

리뷰어가 확인할 최소 조건은 다음과 같다.

1. 범위와 완료 조건이 연결된 이슈와 일치한다.
2. 정상·실패·경계 사례 검증이 있다.
3. 법률 근거, 데이터 계보, 보안 불변조건을 약화하지 않는다.
4. 코드와 문서가 같은 상태를 설명한다.
5. 잔여 위험과 후속 작업이 명시돼 있다.

Draft PR은 방향 확인이나 조기 CI를 위한 상태다. 위 조건을 자체 확인한 뒤 리뷰 준비 상태로 전환한다.
