# RAG 파이프라인 설계

상태: MVP 구현 기준
작성일: 2026-07-13
최종 갱신: 2026-07-18

## 목표

사용자 질문에서 관련 법률 원문을 찾아, 원문 범위를 벗어나지 않는 구조화된 답변과 검증 가능한 인용을 반환한다.

## 문서 모델

최소 메타데이터:

- 안정적 출처 ID와 원문 URL
- 문서 유형, 제목, 발행기관, 관할
- 공포/선고/발행일, 시행일 범위, 수집 시각
- 법령 조·항·호 또는 판례 사건번호/문단 등 위치 식별자
- 원본 콘텐츠 해시, 파서 버전, 청크 버전
- 이전/다음 버전과 관련 문서 관계

청킹은 고정 토큰 길이보다 법률 구조를 우선한다. 법령은 조·항 경계를, 판례는 주문·이유·쟁점과 문단 경계를 보존한다. 너무 긴 단위만 겹침을 가진 하위 청크로 나눈다.

## 검색 단계

1. 질의를 원문과 구조화 필터로 분리한다.
   - `1조2항` 같은 구조 질의는 `제1조/항2`로 정규화하고, Open API 원문의 기존 기호 경로(`제1조/항②`)도 같은 경로로 취급한다.
   - 정확 법령명이 함께 있으면 해당 법령으로 한정하고, 없으면 기준일에 유효한 허용 법령 전체에서 같은 경로를 반환한다.
   - 일반 자연어는 NFKC 정규화 후 질문 표현·조사를 제거하고 한글·영문·숫자 토큰만으로 검색 질의를 재구성한다.
2. 검색 전용 모드는 AI나 임베딩에 의존하지 않는다. 제목·표제·본문을 대상으로 다음 순서로 검색한다.
   - 1단계: 모든 핵심어 일치(`A AND B AND C`)
   - 2단계: 최소 2개 핵심어 후보 풀(`(A AND B) OR (A AND C) OR (B AND C)`). 이 단계 결과는 바로 반환하지 않는다.
   - 3단계: 2단계 후보에 법령명·도메인 앵커를 필수 적용한다. 상위 후보 제한 밖에 앵커 후보가 있을 가능성이 있을 때만 `A AND (B OR C)`를 추가 실행한다.
   - 4단계: 후보가 없으면 근거 부족으로 종료
   - 전체 단일어 `OR` 검색은 실행하지 않는다. 동일한 질의는 다시 DB에 보내지 않고 단계 상태만 기록한다.
3. Terra 모드에서 임베딩을 사용할 수 있을 때만 의미 검색을 추가한다.
4. 결정적 병합 방식으로 후보를 합친다.
5. 재순위 모델은 관련성, 권위, 기준일 적합성, 다양성을 고려한다.
6. 동일 조문·중복 판례 문단을 정리하고 답변 컨텍스트 예산을 배분한다.

초기 수치(`top-k`, 청크 크기, 임계값)는 상수가 아닌 평가 결과로 정한다.

운영 실패를 입력·파싱·질의 변환·코퍼스·검색·의미 검색·생성·결과 단계로 분리한 실제 디버깅 과정과 4단계 검색 개선 결과는 [RAG 검색 시스템 디버깅과 로직 개선](../learning/20-rag-retrieval-debugging-and-improvement.md)에 기록한다.

## 답변 계약

모델 출력은 자유 텍스트가 아니라 다음 논리 구조로 검증한다.

- `summary`: 짧은 결론
- `analysis[]`: 주장, 설명, 연결된 citation ID
- `citations[]`: 문서 ID, 위치, 인용 범위
- `scope`: 관할, 기준일, 자료 범위
- `limitations[]`: 근거 부족, 충돌, 비지원 영역
- `follow_up[]`: 필요한 추가 사실 또는 전문가 확인 제안

검증 실패 시 한 번의 제한된 재생성을 허용하고, 다시 실패하면 근거 목록과 오류 상태만 반환한다.

## 프롬프트 주입 방어

- 수집 문서는 신뢰할 수 없는 데이터로 취급하며 문서 내 명령을 실행하지 않는다.
- 시스템 지침과 원문을 분리하고 허용된 출력 스키마만 받는다.
- 원문에 포함된 URL이나 도구 호출 지시를 자동 실행하지 않는다.
- 모델 출력의 인용 ID가 실제 제공 컨텍스트에 존재하는지 검사한다.

## 캐시와 재현성

캐시 키에는 정규화 질의, 필터, 문서/색인 버전, 검색 설정, 모델 및 프롬프트 버전을 포함한다. 개인정보 가능성이 있는 원문 질의를 공유 캐시 키나 로그에 그대로 저장하지 않는다.

## 실패 모드

| 실패 | 사용자 동작 | 운영 신호 |
|---|---|---|
| 검색 결과 없음 | `no_results` 상태와 조문 경로 부재 또는 일반 근거 불일치 원인을 표시 | zero-result rate 및 원인별 비율 |
| 검색 저장소 장애 | 생성하지 않고 일시 오류 | retrieval availability |
| 모델 장애 | 근거 제목/요약 목록으로 성능 저하 | model error rate |
| 인용 검증 실패 | 답변 차단 또는 축소 | citation validation rate |
| 오래된 색인 | 기준 시점 경고 | index freshness lag |

로그인 질문 이력에는 `input_validation`, `parsing`, `embedding`, `retrieval`, `generation`, `outcome` 단계를 JSONB로 저장한다. 후보 수, 실행 전략, 완화 여부와 실패 상태를 남기되 인증정보·키·오류 전문은 저장하지 않는다. 운영 분석은 `python -m scripts.analyze_question_history --email <email>`의 읽기 전용 JSON 출력으로 수행한다.

Terra 요청이 검색 전용으로 저하되면 질문 응답은 요청 모드와 안전한 `fallback_reason`을 함께 반환한다. 오류 전문이나 계정·키 정보는 공개하지 않는다. `ai_available`은 무과금 상태 확인 API가 아니라 서버 설정과 현재 함수 인스턴스가 관측한 402/429 상태를 나타낸다.

## 결정 기록

- 2026-07-15: 조문 경로가 포함된 질의는 의미 검색보다 경로 일치를 우선한다. 같은 조항 번호가 여러 법령에 존재할 수 있으므로 법령명 미지정 시 임의의 한 법령을 고르지 않는다.
- 2026-07-15: 질문 API의 검색 실패 계약에 `result_status`와 `no_results_reason`을 추가한다. 사용자 문구만 파싱하지 않고 클라이언트가 안정적으로 실패 유형을 표시하기 위함이다.
- 2026-07-15: Terra 요청과 실제 검색 전용 응답의 불일치를 `requested_answer_mode`와 안전한 `fallback_reason`으로 명시한다. 402/429 이후 현재 함수 인스턴스는 Terra를 차단하지만, 서버리스 인스턴스 간 공유 차단은 영속 런타임 플래그 도입 전까지 보장하지 않는다.
- 2026-07-18: AI 비활성 상태를 0건의 원인으로 취급하지 않는다. 키워드 검색은 독립 실행하고, 자연어 정규화와 제목·표제·본문 검색 및 0건 완화를 적용한다.
- 2026-07-18: 실패 원인을 재현할 수 있도록 인증 질문 이력에 단계별 진단을 보존한다.
- 2026-07-18: 전체 `OR` 완화는 노이즈가 커서 제거하고, 모든 핵심어·최소 2개·앵커 필수·근거 부족의 4단계 계약을 사용한다. 검색 전용은 AI를 호출하지 않으며 Terra 모드의 임베딩·생성은 유지한다.
