# Auto Generating Migrations

> 생성 명령: Codex를 사용한 사용자 제공 원문의 한국어 번역
> 기준 시점: 2026-07-14
> 출처: 사용자 제공 Alembic `Auto Generating Migrations` 문서 캡처 · [공식 문서](https://alembic.sqlalchemy.org/en/latest/autogenerate.html)
> 번역 원칙: 코드, 명령, class·function·parameter 이름과 중요 전문용어는 원문 그대로 유지하고 어려운 용어에만 한국어 해석을 괄호로 덧붙였다. 내용을 생략하거나 요약하지 않았다.

Alembic은 `alembic.ini` 파일의 `sqlalchemy.url`이 가리키는 database의 현재 schema 상태를 확인하고, 이를 application 안의 table metadata, 즉 제안된 schema를 정의하는 ORM과 비교할 수 있습니다. 그리고 이 비교를 바탕으로 “명백한” migration을 생성할 수 있습니다. 이는 `alembic revision` command에 `--autogenerate` option을 사용하여 수행하며, 이 option은 이른바 candidate migration(후보 migration)을 새 migration file에 넣습니다. 우리는 필요에 따라 이를 직접 검토하고 수정한 다음 정상 절차를 진행합니다.

`autogenerate`를 사용하려면 먼저 `env.py`를 수정하여 target을 포함하는 table metadata object에 접근할 수 있게 해야 합니다. Application에 `myapp.mymodel`이라는 declarative base가 있다고 가정해 봅시다. 이 base에는 database를 정의하는 `Table` object를 포함하는 `MetaData` object가 있습니다. 이 object가 `env.py`에서 load되는지 확인한 다음, `target_metadata` argument를 통해 `EnvironmentContext.configure()`에 전달합니다. Generic template에서 사용하는 `env.py` sample script에는 편의를 위해 이미 파일 위쪽에 variable declaration이 있으며, 여기서 `None`을 우리의 `MetaData`로 바꿉니다. 다음 코드로 시작합니다.

```python
# add your model's MetaData object here
# for 'autogenerate' support
# from myapp import mymodel
# target_metadata = mymodel.Base.metadata
target_metadata = None
```

이를 다음과 같이 변경합니다.

```python
from myapp.mymodel import Base
target_metadata = Base.metadata
```

> **Note**
> 위 예제는 `alembic init`을 호출할 때 기본으로 만들어지는 것과 같은 generic Alembic `env.py` template을 말하며, `multidb` 같은 특수 목적 template을 말하는 것이 아닙니다. `autogenerate` metadata가 어디에서 어떻게 설정되는지에 관한 구체적인 지침은 `env.py` script 안의 source code와 comment를 직접 확인하십시오.

Script 뒤쪽의 `run_migrations_online()`을 보면 `EnvironmentContext.configure()`에 전달되는 directive를 확인할 수 있습니다.

```python
def run_migrations_online():
    engine = engine_from_config(
                config.get_section(config.config_ini_section), prefix='sqlalchemy.')

    with engine.connect() as connection:
        context.configure(
                    connection=connection,
                    target_metadata=target_metadata
                    )

        with context.begin_transaction():
            context.run_migrations()
```

이제 `alembic revision` command와 `--autogenerate` option을 함께 사용할 수 있습니다. `MetaData`에는 `account` table의 definition이 있지만 database에는 없다고 가정해 봅시다. 다음과 같은 output을 얻게 됩니다.

```console
$ alembic revision --autogenerate -m "Added account table"
INFO [alembic.context] Detected added table 'account'
Generating /path/to/foo/alembic/versions/27c6a30d7c24.py...done
```

그런 다음 `27c6a30d7c24.py` file을 열어보면 기초적인 migration이 이미 들어 있습니다.

```python
"""empty message

Revision ID: 27c6a30d7c24
Revises: None
Create Date: 2011-11-08 11:40:27.089406

"""

# revision identifiers, used by Alembic.
revision = '27c6a30d7c24'
down_revision = None

from alembic import op
import sqlalchemy as sa

def upgrade():
    ### commands auto generated by Alembic - please adjust! ###
    op.create_table(
    'account',
    sa.Column('id', sa.Integer()),
    sa.Column('name', sa.String(length=50), nullable=False),
    sa.Column('description', sa.VARCHAR(200)),
    sa.Column('last_transaction_date', sa.DateTime()),
    sa.PrimaryKeyConstraint('id')
    )
    ### end Alembic commands ###

def downgrade():
    ### commands auto generated by Alembic - please adjust! ###
    op.drop_table("account")
    ### end Alembic commands ###
```

물론 migration은 아직 실제로 실행되지 않았습니다. 평소의 `upgrade` command를 통해 실행합니다. 또한 migration file에 들어가 필요에 따라 수정해야 합니다. 여기에는 directive 자체를 조정하는 일뿐 아니라, 해당 directive가 의존할 수 있는 다른 directive, 특히 create·alter·drop 사이에 수행해야 하는 data change를 추가하는 일도 포함됩니다.

## What does Autogenerate Detect (and what does it not detect?)

Alembic에 관한 사용자 문제의 대부분은 `autogenerate`가 어떤 변경을 안정적으로 감지할 수 있고 어떤 변경을 감지할 수 없는지, 그리고 감지한 내용을 Python code로 어떻게 render하는지를 중심으로 발생합니다. `autogenerate`가 완벽하도록 설계된 기능이 아니라는 점에 주목하는 것이 매우 중요합니다. `autogenerate`가 만든 candidate migration은 항상 직접 검토하고 수정해야 합니다. Release가 이어지면서 이 기능은 점점 더 포괄적이고 오류가 적어지고 있지만, 현재의 limitation(제약)을 알아두어야 합니다.

`autogenerate`는 다음을 감지합니다.

- Table 추가와 제거
- Column 추가와 제거
- Column의 `nullable` 상태 변경
- Index와 명시적으로 이름을 지정한 unique constraint의 기본적인 변경
- Foreign key constraint의 기본적인 변경

`autogenerate`는 선택적으로 다음을 감지할 수 있습니다.

- Column type 변경. `EnvironmentContext.configure.compare_type` parameter를 `False`로 설정하지 않는 한 기본적으로 실행됩니다. 기본 구현은 `Numeric`과 `String` 사이의 변경 같은 주요 변경을 안정적으로 감지하며, `Boolean` 같은 SQLAlchemy의 “generic” type에서 생성되는 type도 처리합니다. Length와 precision처럼 두 type이 공유하는 argument도 비교합니다. Metadata type이나 database type 중 하나가 다른 쪽에 없는 추가 argument를 가진 경우에는 비교하지 않습니다. 예를 들어 한 numeric type에는 `scale`이 있지만 다른 type에는 없는 경우, 이는 backing database가 그 값을 지원하지 않거나 metadata가 지정하지 않은 default를 database가 보고한 것으로 간주합니다.

Type comparison logic은 완전히 확장할 수 있습니다. 자세한 내용은 `Comparing Types`를 참조하십시오.

- Server default 변경. `EnvironmentContext.configure.compare_server_default` parameter를 `True` 또는 custom callable function으로 설정하면 실행됩니다. 이 기능은 단순한 경우에는 잘 작동하지만 항상 정확한 결과를 만들 수 있는 것은 아닙니다. PostgreSQL backend는 동등성을 판단하기 위해 database에서 “detected” value와 “metadata” value를 실제로 실행합니다. Target schema에서 먼저 시험할 수 있도록 이 기능은 기본적으로 꺼져 있습니다. Type comparison과 마찬가지로 callable을 전달하여 customize할 수 있습니다. 자세한 내용은 해당 function 문서를 참조하십시오.

`autogenerate`는 다음을 감지할 수 없습니다.

- Table name 변경. 이는 서로 다른 두 table의 add/drop으로 출력되므로, name change가 되도록 직접 수정해야 합니다.
- Column name 변경. Table name 변경과 마찬가지로 column add/drop pair로 감지되며, 이는 name change와 전혀 같지 않습니다.
- 익명으로 이름이 지정된 constraint. 예를 들어 `UniqueConstraint('col1', 'col2', name="my_name")`처럼 constraint에 이름을 지정하십시오. Constraint의 automatic naming scheme을 설정하는 배경은 `The Importance of Naming Constraints` section을 참조하십시오.
- `ENUM`을 직접 지원하지 않는 backend에서 생성되는 `Enum` 같은 특별한 SQLAlchemy type. 그 이유는 지원하지 않는 database에서 이 type을 표현한 `CHAR + CHECK constraint`가 어떤 종류의 `CHAR + CHECK`도 될 수 있기 때문입니다. SQLAlchemy가 이것이 실제로 `ENUM`이라고 판단하는 것은 추측일 뿐이며, 일반적으로 좋지 않은 생각입니다. 여기에 자체 “guessing” function을 구현하려면 `sqlalchemy.events.DDLEvents.column_reflect()` event를 사용하십시오. `CHAR` 또는 target type이 reflection(리플렉션, DB 구조 읽기)될 때 이를 감지하고, 해당 type의 의도가 알려져 있다면 `ENUM` 또는 원하는 type으로 변경할 수 있습니다. `sqlalchemy.events.DDLEvents.after_parent_attach()`는 `autogenerate` 과정에서 원하지 않는 `CHECK constraint`를 가로채 연결 해제하는 데 사용할 수 있습니다.

`autogenerate`는 현재 감지할 수 없지만 향후 감지하게 될 항목은 다음과 같습니다.

- 독립적으로 존재하는 일부 constraint의 추가와 제거는 지원되지 않을 수 있습니다. 여기에는 `PRIMARY KEY`, `EXCLUDE`, `CHECK`가 포함됩니다. 이들은 `autogenerate` detection system에 반드시 구현되어 있는 것은 아니며, 이를 지원하는 SQLAlchemy dialect에서도 지원되지 않을 수 있습니다.
- Sequence 추가와 제거는 아직 구현되지 않았습니다.

### Notable 3rd-party libraries that extend the built-in Alembic autogenerate functionality

- `alembic-utils`: PostgreSQL function, view, trigger 등에 대한 `autogenerate` 지원을 추가하는 library입니다.
- `alembic-postgresql-enum`: PostgreSQL에서 `Enum`의 생성, 변경, 삭제에 대한 `autogenerate` 지원을 추가하는 library입니다.

## Autogenerating Multiple MetaData collections

Application에 여러 `MetaData` collection이 관련된 경우 `target_metadata` collection을 sequence로 정의할 수도 있습니다.

```python
from myapp.mymodel1 import Model1Base
from myapp.mymodel2 import Model2Base
target_metadata = [Model1Base.metadata, Model2Base.metadata]
```

`autogenerate` 과정에서는 `MetaData` collection sequence를 순서대로 참조합니다. 각 `MetaData`에는 고유한 table key가 있어야 합니다. 예를 들어 “key”는 table name과 schema의 조합입니다. 두 `MetaData` object가 같은 schema/name 조합의 table을 포함하면 오류가 발생합니다.

## Controlling What to be Autogenerated

`autogenerate` 과정은 현재 사용 중인 database connection이 가리키는 database 안의 모든 table object를 scan합니다.

Target database connection에서 scan하는 object 목록은 다음과 같습니다.

- Database connection이 현재 가리키는 “default” schema
- `EnvironmentContext.configure.include_schemas`를 `True`로 설정한 경우, `Inspector`의 `get_schema_names()` method가 반환하는 이름에 해당하는 모든 non-default “schema”. SQLAlchemy 문서 `Specifying the Schema Name`은 “schema” concept를 자세히 설명합니다.
- 각 “schema” 안에서는 `Inspector`의 `get_table_names()` method를 사용하여 존재하는 모든 table을 scan합니다.
- 각 “table” 안에서는 column과 일부 constraint를 포함하여 각 `Table` construct의 하위 object 대부분을 scan합니다. 이 과정은 최종적으로 `get_columns()`, `get_indexes()`, `get_unique_constraints()`, `get_foreign_keys()`를 포함한 `Inspector` method를 사용합니다. 이 글을 쓰는 시점에는 `CHECK constraint`와 primary key constraint가 아직 포함되지 않습니다.

**See also**

- `Specifying the Schema Name`: SQLAlchemy가 schema name을 해석하는 방법에 대한 심층 소개
- `Remote-Schema Table Introspection and PostgreSQL search_path`: PostgreSQL database에 특화된 중요 참고사항

## Omitting Schema Names from the Autogenerate Process

위 database object 집합은 일반적으로 하나의 `MetaData` object 내용과 비교됩니다. 특히 `EnvironmentContext.configure.include_schemas` flag가 활성화되어 있을 때는 원하지 않는 “schema”를 걸러내는 것이 중요합니다. 일부 database backend에서는 이 목록이 존재하는 모든 database의 목록일 수도 있습니다. 이 filtering은 `EnvironmentContext.configure.include_name` hook을 사용하는 것이 가장 좋습니다. 이 hook은 특정 schema name을 포함할 것인지 나타내는 boolean `true`/`false`를 반환할 수 있는 callable을 제공합니다.

```python
def include_name(name, type_, parent_names):
    if type_ == "schema":
        # note this will not include the default schema
        return name in ["schema_one", "schema_two"]
    else:
        return True

context.configure(
    # ...
    include_schemas = True,
    include_name = include_name
)
```

위 예제에서는 schema name 목록을 처음 가져올 때 위 `include_name` function을 통해 이름을 filter하므로, `"schema_one"`과 `"schema_two"`라는 이름의 schema만 `autogenerate` 과정에서 고려합니다.

명시적인 schema를 지정하지 않았을 때 database connection이 가리키는 schema인 default schema를 포함하려면 hook에 전달되는 name은 `None`입니다. 위 예제를 변경하여 default schema도 포함하려면 `None`도 함께 비교합니다.

```python
def include_name(name, type_, parent_names):
    if type_ == "schema":
        # this **will* include the default schema
        return name in [None, "schema_one", "schema_two"]
    else:
        return True

context.configure(
    # ...
    include_schemas = True,
    include_name = include_name
)
```

## Omitting Table Names from the Autogenerate Process

`EnvironmentContext.configure.include_name` hook은 고려할 target database의 table name을 제한하는 데에도 가장 적합합니다. Target database에 `MetaData`에 포함되지 않은 table이 많으면, `autogenerate` 과정은 일반적으로 이를 제거해야 할 불필요한 table로 간주하여 각각에 대해 `Operations.drop_table()` operation을 생성합니다. 이를 방지하려면 `EnvironmentContext.configure.include_name` hook을 사용해 각 name이 `MetaData` object의 `tables` collection 안에 있는지 검색하고, 없는 name이 포함되지 않게 할 수 있습니다.

```python
target_metadata = MyModel.metadata

def include_name(name, type_, parent_names):
    if type_ == "table":
        return name in target_metadata.tables
    else:
        return True

context.configure(
    # ...
    target_metadata = target_metadata,
    include_name = include_name,
    include_schemas = False
)
```

위 예제는 default schema에 존재하는 table name으로만 제한됩니다. Schema-qualified table name(스키마가 붙은 table name)도 `MetaData` collection에서 검색하려면, non-default schema에 존재하는 table은 `<schemaname>.<tablename>` 형식의 name 아래에 존재합니다. `EnvironmentContext.configure.include_name` hook은 `parent_names` dictionary의 `"schema_name"` key를 통해 table name별로 이 schema name을 제공합니다. 이 key는 현재 고려 중인 schema name을 가리키며, database connection의 default schema라면 `None`입니다.

```python
# example fragment

if parent_names["schema_name"] is None:
    return name in target_metadata.tables
else:
    # build out schema-qualified name explicitly...
    return (
        "%s.%s" % (parent_names["schema_name"], name) in
        target_metadata.tables
    )
```

더 간단하게는 `parent_names` dictionary가 이미 점으로 연결해 만든 name을 `"schema_qualified_table_name"` key에 포함합니다. Default schema의 table도 점을 생략한 올바른 형식으로 제공합니다. 따라서 schema를 지원하면서 table을 제외하는 전체 예제는 다음과 같습니다.

```python
target_metadata = MyModel.metadata

def include_name(name, type_, parent_names):
    if type_ == "schema":
        return name in [None, "schema_one", "schema_two"]
    elif type_ == "table":
        # use schema_qualified_table_name directly
        return (
            parent_names["schema_qualified_table_name"] in
            target_metadata.tables
        )
    else:
        return True

context.configure(
    # ...
    target_metadata = target_metadata,
    include_name = include_name,
    include_schemas = True
)
```
고려 중인 name이 특정 table에 속한 column 또는 constraint object의 name인 경우, `parent_names` dictionary에는 `"table_name"` key도 포함됩니다.

`EnvironmentContext.configure.include_name` hook은 reflection된 object만 대상으로 하며, target `MetaData` collection 안에 있는 object는 대상으로 하지 않습니다. `MetaData`와 reflection된 object를 모두 포함하는 더 세밀한 rule에는 다음 section에서 설명하는 `EnvironmentContext.configure.include_object` hook이 더 적합합니다.

## Omitting Based on Object

`EnvironmentContext.configure.include_object` hook은 reflection되는 `Table` object와 그 안의 element를 바탕으로 object level의 포함·제외 rule을 제공합니다. 이 hook은 local `MetaData` collection과 target database 양쪽의 object를 제한하는 데 사용할 수 있습니다. 다만 database의 object를 대상으로 판단할 때는 해당 object를 완전히 reflection한 뒤이므로, 제외할 object가 많다면 비용이 클 수 있다는 limitation이 있습니다. 아래 예제는 `info` dictionary에 사용자가 정의한 `skip_autogenerate` flag가 들어 있는 `Column` object의 변경을 건너뛰는 세밀한 rule을 보여줍니다.

```python
def include_object(object, name, type_, reflected, compare_to):
    if (type_ == "column" and
        not reflected and
        object.info.get("skip_autogenerate", False)):
        return False
    else:
        return True

context.configure(
    # ...
    include_object = include_object
)
```

## Comparing and Rendering Types

Migration script에서 Python 기반 type object를 비교하고 render하는 `autogenerate` 동작은 어려운 문제를 제시합니다. Script에 render해야 할 type은 SQLAlchemy의 type과 user-defined type을 포함하여 매우 다양하기 때문입니다. 이 작업을 돕기 위한 몇 가지 option이 제공됩니다.

### Controlling the Module Prefix

Type을 render할 때는 비교적 적은 수의 import만으로 사용할 수 있도록 module prefix를 붙여 생성합니다. Prefix에 관한 rule은 datatype의 종류와 configuration setting을 기반으로 합니다. 예를 들어 Alembic이 SQLAlchemy type을 render할 때는 기본적으로 type name 앞에 `sa.` prefix를 붙입니다.

```python
Column("my_column", sa.Integer())
```

`sa.` prefix 사용 여부는 `EnvironmentContext.configure.sqlalchemy_module_prefix` 값을 변경하여 제어할 수 있습니다.

```python
def run_migrations_online():
    # ...

    context.configure(
                connection=connection,
                target_metadata=target_metadata,
                sqlalchemy_module_prefix="sqla.",
                # ...
                )

    # ...
```

어느 경우든 `sa.` prefix 또는 원하는 다른 prefix를 `script.py.mako`의 imports section에도 포함해야 합니다. 이 section의 default 역시 `import sqlalchemy as sa`입니다.

User-defined type, 즉 `sqlalchemy.` module namespace 안에 들어 있지 않은 custom type의 경우 Alembic은 기본적으로 custom type의 `__module__` value를 사용합니다.

```python
Column("my_column", myapp.models.utils.types.MyCustomType())
```

위 type의 import도 수동으로 추가하거나 `script.py.mako`에 추가하여 migration 안에 존재하게 해야 합니다.

위 custom type은 `__module__`을 직접 사용하므로 이름이 길고 번거롭습니다. 또한 type이 많아지면 이를 처리하기 위해 많은 import가 필요하다는 뜻이기도 합니다. 이런 이유로 migration script에서 사용하는 user-defined type은 하나의 module에서 사용할 수 있게 만드는 것을 권장합니다. 그 module을 `myapp.migration_types`라고 하겠습니다.

```python
# myapp/migration_types.py

from myapp.models.utils.types import MyCustomType
```

먼저 `script.py.mako`에 `migration_types` import를 추가할 수 있습니다.

```python
from alembic import op
import sqlalchemy as sa
import myapp.migration_types
${imports if imports else ""}
```

그런 다음 `EnvironmentContext.configure.user_module_prefix` option을 사용하여 고정 prefix를 제공함으로써 Alembic의 `__module__` 사용을 override합니다.

```python
def run_migrations_online():
    # ...

    context.configure(
                connection=connection,
                target_metadata=target_metadata,
                user_module_prefix="myapp.migration_types.",
                # ...
                )

    # ...
```

이제 다음과 같은 migration을 얻게 됩니다.

```python
Column("my_column", myapp.migration_types.MyCustomType())
```

나중에 application을 refactor하여 `MyCustomType`을 다른 곳으로 옮기더라도, 모든 migration script에서 instance를 찾아 바꾸는 대신 `myapp.migration_types` module만 수정하면 됩니다.

### Affecting the Rendering of Types Themselves

Alembic이 SQLAlchemy와 user-defined type construct를 Python code로 생성할 때 사용하는 방식은 평범한 `__repr__()`입니다. SQLAlchemy built-in type의 `__repr__()`은 대부분 Python과 호환되는 constructor call을 충실하게 render하지만, 일부 예외가 있습니다. 특히 constructor가 pickling function처럼 `__repr__()`과 호환되지 않는 argument를 받는 경우가 그렇습니다.

Migration script에 render할 custom type을 만들 때는 해당 type의 constructor를 충실하게 재현하는 `__repr__()`을 명시적으로 제공해야 하는 경우가 많습니다. 이를 `EnvironmentContext.configure.user_module_prefix`와 함께 사용하는 것으로 보통 충분합니다. 하지만 추가 동작이 필요하다면 `EnvironmentContext.configure.render_item` option이라는 더 포괄적인 hook이 있습니다. 이 hook을 사용하면 `env.py` 안에 callable function을 제공하여 module prefix를 포함한 type의 render 방식을 완전히 제어할 수 있습니다.

```python
def render_item(type_, obj, autogen_context):
    """Apply custom rendering for selected items."""

    if type_ == 'type' and isinstance(obj, MySpecialType):
        return "mypackage.%r" % obj

    # default rendering for other objects
    return False

def run_migrations_online():
    # ...

    context.configure(
                connection=connection,
                target_metadata=target_metadata,
                render_item=render_item,
                # ...
                )

    # ...
```

위 예제에서는 `MySpecialType`이 적절한 `__repr__()` method를 포함하도록 해야 합니다. 이 method는 `"%r"`에 대해 object를 호출할 때 실행됩니다.

`EnvironmentContext.configure.render_item`에 사용하는 callable은 migration script에 import를 추가할 수도 있습니다. 전달되는 `AutogenContext`에는 `AutogenContext.imports`라는 data member가 있으며, 이는 새 import를 추가할 수 있는 Python `set()`입니다. 예를 들어 `MySpecialType`이 `mymodel.types`라는 module 안에 있다면 해당 type을 만났을 때 import를 추가할 수 있습니다.

```python
def render_item(type_, obj, autogen_context):
    """Apply custom rendering for selected items."""

    if type_ == 'type' and isinstance(obj, MySpecialType):
        # add import for this type
        autogen_context.imports.add("from mymodel import types")
        return "types.%r" % obj

    # default rendering for other objects
    return False
```

완성된 migration script는 `${imports}` expression이 사용되는 곳에 import를 포함하여 다음과 같은 output을 만듭니다.

```python
from alembic import op
import sqlalchemy as sa
from mymodel import types

def upgrade():
    op.add_column('sometable', Column('mycolumn', types.MySpecialType()))
```

## Comparing Types

Default type comparison logic은 SQLAlchemy built-in type과 기본적인 user-defined type에서 작동합니다. 이 logic은 기본적으로 활성화되어 있습니다. `EnvironmentContext.configure.compare_type`을 `False`로 설정하여 비활성화할 수 있습니다.

```python
context.configure(
    # ...
    compare_type = False
)
```

Version 1.12.0에서 변경됨: `EnvironmentContext.configure.compare_type`의 default value가 `True`로 변경되었습니다.

> **Note**
> End user가 확장할 수 있는 default type comparison logic은 현재 Alembic version 1.4.0을 기준으로 column에 대해 생성된 SQL을 비교하는 방식으로 작동합니다. 이 작업은 두 단계로 이루어집니다.

먼저 각 column의 `VARCHAR` 또는 `TEXT` 같은 outer type을 비교합니다. 일부 database는 type을 다른 type으로 변환하여 지원하므로 dialect 구현에는 동등한 것으로 간주하는 synonym(동의 type)이 있을 수 있습니다. 예를 들어 모든 backend에서 `NUMERIC`과 `DECIMAL`은 동등한 것으로 간주하며, Oracle backend에서는 여기에 `BIGINT`, `INTEGER`, `NUMBER`, `SMALLINT`도 동등한 type 목록으로 추가합니다.

다음으로 string의 length, numeric의 precision value, enumeration 안의 element 같은 type 내부 argument를 비교합니다. **양쪽** column에 argument가 있고 그 값이 다르면 변경을 감지합니다. 한 column은 default만 사용하고 다른 column에 argument가 있다면 Alembic은 비교를 시도하지 않습니다. Database backend가 설정하는 default value를 false positive(거짓 양성) 없이 감지하기 어렵기 때문입니다.

또는 `EnvironmentContext.configure.compare_type` parameter에 callable function을 전달하여 특별한 user-defined type 같은 경우를 위한 custom type comparison logic을 구현할 수 있습니다.

```python
def my_compare_type(context, inspected_column,
            metadata_column, inspected_type, metadata_type):
    # return False if the metadata_type is the same as the inspected_type
    # or None to allow the default implementation to compare these
    # types. a return value of True means the two types do not
    # match and should result in a type change operation.
    return None

context.configure(
    # ...
    compare_type = my_compare_type
)
```

위 코드에서 `inspected_column`은 `sqlalchemy.engine.reflection.Inspector.reflect_table()`이 반환한 `sqlalchemy.schema.Column`이고, `metadata_column`은 local model environment의 `sqlalchemy.schema.Column`입니다. `None`을 반환하면 default type comparison이 계속 진행됩니다.

## Applying Post Processing and Python Code Formatters to Generated Revisions

`alembic revision` command가 생성하는 revision script는 선택적으로 일련의 post-production function(후처리 함수)을 거칠 수 있습니다. 이 function은 revision command가 실행되는 범위 안에서 Alembic이 생성한 Python source code를 분석하거나 다시 작성할 수 있습니다. 이 기능의 주된 목적은 Alembic이 revision file을 생성할 때 `Black`이나 `autopep8` 같은 code-formatting tool, custom formatting function, linter function을 실행하는 것입니다. Hook은 원하는 수만큼 설정할 수 있으며, 새로 생성된 file의 path와 configuration option을 전달받아 순서대로 실행됩니다.

설정된 post write hook은 `autogenerate` 기능 사용 여부와 관계없이 생성된 revision file을 대상으로 실행됩니다.

> **Note**
> Alembic의 post write system은 file이 Git repository에 commit될 때 source file을 다시 format하도록 git hook을 설정하는 `pre-commit` tool에서 일부 영감을 받았습니다. `pre-commit`도 Alembic revision file에 이 역할을 수행하여 commit 시점에 code formatter를 적용할 수 있습니다. Alembic의 post write hook은 commit 시점이 아니라 file 생성 직후 즉시 format할 수 있다는 점에서 유용하며, `pre-commit`을 사용하지 않으려는 project에서도 유용할 수 있습니다.

### Basic Post Processor Configuration

현재 관련 template의 `alembic.ini`와 `pyproject.toml` template sample에는 새로 생성된 file path를 대상으로 code-formatting tool이나 linter 같은 다른 tool을 실행하는 방법을 보여주는 주석 처리된 configuration이 들어 있습니다. 다음은 `alembic.ini` file의 예입니다.

```ini
[post_write_hooks]

# format using "black"
hooks=black

black.type = console_scripts
black.entrypoint = black
black.options = -l 79 REVISION_SCRIPT_FILENAME
```

같은 예제를 `pyproject.toml` file에 설정하면 다음과 같습니다.

```toml
[[tool.alembic.post_write_hooks]]

# format using "black"
name = "black"
type = "console_scripts"
entrypoint = "black"
options = "-l 79 REVISION_SCRIPT_FILENAME"
```

위에서는 hook을 `"black"`이라는 label을 가진 하나의 post write hook으로 설정합니다. 이 label은 임의로 정한 것임에 유의하십시오. 그 다음 `"black"` post write hook의 configuration을 정의합니다. 이 configuration에는 다음이 포함됩니다.

`type`: 실행할 hook의 type입니다. Alembic에는 세 가지 hook runner가 있습니다.

- `"console_scripts"`: `subprocess.run()`을 사용하여 revision file에 대해 별도의 Python script를 호출하는 Python function입니다.
- `"exec"`: `subprocess.run()`을 사용하여 임의의 binary를 실행합니다.
- `"module"`: `subprocess.run()`을 사용하여 Python module을 직접 호출합니다.

Custom hook function에서는 이 configuration variable이 해당 custom hook이 register된 name을 가리킵니다. 예제는 다음 section을 참조하십시오.

Version 1.12에서 추가됨: 새로운 `exec` runner 추가
Version 1.16.3에서 추가됨: 새로운 `module` runner 추가

다음 configuration option은 `"console_scripts"` hook runner에만 적용됩니다.

`entrypoint`: console script를 정의하는 데 사용한 `setuptools` entrypoint의 name입니다. 표준 Python console script 범위에서 이 name은 일반적으로 code formatting tool을 실행할 때 사용하는 shell command의 name과 일치합니다. 여기서는 `black`입니다.

다음 configuration option은 `"exec"` hook runner에만 적용됩니다.

`executable`: 호출할 executable의 name입니다. `$PATH`에서 검색할 일반 executable name이거나, path interception(경로 가로채기) 문제를 피하기 위한 full pathname일 수 있습니다.

다음 option은 `"console_scripts"`와 `"exec"`가 모두 지원합니다.

`options`: code formatting tool에 전달할 command-line option 한 줄입니다. 여기서는 `black /path/to/revision.py -l 79` command를 실행하려고 합니다. 기본적으로 revision path가 첫 번째 argument로 배치됩니다. 다른 위치를 지정하려면 뒤의 예제처럼 `REVISION_SCRIPT_FILENAME` token을 사용할 수 있습니다.

> **Note**
> Script가 input file을 제자리에서 다시 작성하도록 option을 제공해야 합니다. 예를 들어 `autopep8`을 실행할 때는 `--in-place` option을 제공해야 합니다.

```ini
[post_write_hooks]
hooks = autopep8
autopep8.type = console_scripts
autopep8.entrypoint = autopep8
autopep8.options = --in-place REVISION_SCRIPT_FILENAME
```

`cwd`: code processing tool을 실행할 선택적인 working directory입니다.

`alembic revision -m "rev1"`을 실행하면 `black` tool의 output이 다음과 같이 나타납니다.

```console
$ alembic revision -m "rev1"
  Generating /path/to/project/versions/481b13bc369a_rev1.py ... done
  Running post write hook "black" ...
reformatted /path/to/project/versions/481b13bc369a_rev1.py
All done! ✨ 🍰 ✨
1 file reformatted.
  done
```

Hook은 순서대로 실행할 hook runner에 대응하는 name 목록으로 지정할 수도 있습니다. 예를 들어 Alembic 작성자가 만든 `zimports` import rewriting tool을 `black` 이후에 실행할 수도 있습니다. `alembic.ini` configuration은 다음과 같습니다.

```ini
[post_write_hooks]

# format using "black", then "zimports"
hooks=black, zimports

black.type = console_scripts
black.entrypoint = black
black.options = -l 79 REVISION_SCRIPT_FILENAME

zimports.type = console_scripts
zimports.entrypoint = zimports
zimports.options = --style google REVISION_SCRIPT_FILENAME
```

동등한 `pyproject.toml` configuration은 다음과 같습니다.

```toml
# format using "black", then "zimports"

[[tool.alembic.post_write_hooks]]
name = "black"
type="console_scripts"
entrypoint = "black"
options = "-l 79 REVISION_SCRIPT_FILENAME"

[[tool.alembic.post_write_hooks]]
name = "zimports"
type="console_scripts"
entrypoint = "zimports"
options = "--style google REVISION_SCRIPT_FILENAME"
```

위 configuration을 사용하면 새로 생성된 revision file은 먼저 `"black"` tool로 처리된 다음 `"zimports"` tool로 처리됩니다.

또는 다음과 같이 `pre-commit` 자체를 실행할 수 있습니다.

```ini
[post_write_hooks]

hooks = pre-commit

pre-commit.type = console_scripts
pre-commit.entrypoint = pre-commit
pre-commit.options = run --files REVISION_SCRIPT_FILENAME
pre-commit.cwd = %(here)s
```

마지막 줄은 command를 어느 위치에서 호출했는지와 관계없이 `.pre-commit-config.yaml` file을 항상 찾을 수 있도록 돕습니다.

## Writing Custom Hooks as Python Functions

이전 section에서는 Alembic이 `console_scripts`라는 이름으로 제공하는 post write hook을 통해 command-line code formatter를 실행하는 방법을 보여주었습니다. 이 hook은 실제로 registration function을 사용해 해당 name으로 register된 Python function이며, 같은 registration function으로 다른 종류의 hook도 register할 수 있습니다.

이를 설명하기 위해 생성된 code에서 네 개의 space 대신 tab을 사용하도록 다시 작성하려는 짧은 Python function의 예를 사용하겠습니다. 단순화를 위해 이 function을 `env.py` file 안에 직접 둘 수 있는 방법을 보여줍니다. Function은 `write_hooks.register()` decorator를 사용하여 선언하고 register합니다.

```python
from alembic.script import write_hooks
import re

@write_hooks.register("spaces_to_tabs")
def convert_spaces_to_tabs(filename, options):
    lines = []
    with open(filename) as file_:
        for line in file_:
            lines.append(
                re.sub(
                    r"^(    )+",
                    lambda m: "\t" * (len(m.group(1)) // 4),
                    line
                )
            )
    with open(filename, "w") as to_write:
        to_write.write("".join(lines))
```

새 `"spaces_to_tabs"` hook은 `alembic.ini`에서 다음과 같이 설정할 수 있습니다.

```ini
[alembic]

# ...

# ensure the revision command loads env.py
revision_environment = true

[post_write_hooks]

hooks = spaces_to_tabs

spaces_to_tabs.type = spaces_to_tabs
```

`alembic revision`을 실행하면 항상 `env.py` file이 load되고 custom `"spaces_to_tabs"` function이 register된 뒤 새로 생성된 file path를 대상으로 실행됩니다.

```console
$ alembic revision -m "rev1"
  Generating /path/to/project/versions/481b13bc369a_rev1.py ... done
  Running post write hook "spaces_to_tabs" ...
  done
```

## Running Alembic Check to test for new upgrade operations

Code를 개발할 때 일련의 code change가 database model에 실질적인 변경을 만들었고, 그 결과 새 revision을 생성해야 하는지 알아내는 것이 유용합니다. 이를 자동화하기 위해 Alembic은 `alembic check` command를 제공합니다. 이 command는 revision file이 생성되기 직전까지 `alembic revision --autogenerate`와 같은 process를 수행하지만 새 file은 생성하지 않습니다. 대신 새 revision에 render될 새로운 operation을 감지하면 error code와 message를 반환합니다. 감지하지 않으면 success code와 message를 반환합니다. `alembic check`가 success code를 반환하면 `alembic revision --autogenerate`가 빈 migration만 만들 것이므로 실행할 필요가 없음을 의미합니다.

`alembic check`를 CI system과 on-commit scheme에 포함하여 들어오는 code가 새 revision 생성을 요구하지 않는지 확인할 수 있습니다. 아래 예제는 새 operation을 감지한 check를 보여줍니다.

```console
$ alembic check
FAILED: New upgrade operations detected: [
  ('add_column', None, 'my_table', Column('data', String(), table=<my_table>)),
  ('add_column', None, 'my_table', Column('newcol', Integer(), table=<my_table>))]
```

반대로 새로운 operation을 감지하지 않으면 다음과 같습니다.

```console
$ alembic check
No new upgrade operations detected.
```

Version 1.9.0에서 추가됨.

> **Note**
> `alembic check` command는 `alembic revision --autogenerate` process와 같은 model comparison process를 사용합니다. 이는 `EnvironmentContext.configure.compare_type`과 `EnvironmentContext.configure.compare_server_default` 같은 parameter가 평소와 마찬가지로 적용되며, `autogenerate` detection의 limitation도 `alembic check`를 실행할 때 동일하다는 뜻입니다.
