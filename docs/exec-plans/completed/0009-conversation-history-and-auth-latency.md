# 실행 계획 0009: 연속 대화, 이력 페이지네이션, 인증 지연 개선

상태: 완료  
작성일: 2026-07-18  
소유자: Codex

## 목적과 사용자 결과

한 화면에서 질문과 답변을 연속으로 쌓고, 저장된 대화는 첫 질문 제목으로 다시 열 수 있게 한다. 긴 대화는 리소스를 무제한 사용하지 않도록 400개 메시지 경계에서 새 대화로 자동 전환한다. 로그인 상태와 질문 이력은 초기 화면을 불필요하게 막지 않고, 이력은 필요한 페이지와 상세만 비동기로 읽는다.

## 범위와 비범위

범위는 대화/턴 저장 계약, 사이드바 대화 요약, cursor 페이지네이션, 상세 지연 로딩, 삭제, 요청 취소 UI, 인증 호출 중복 제거, 이미지에 표시된 문구·외부 링크 제거다. LLM 대화형 추론 컨텍스트, 과거 대화 요약·압축, 스트리밍 생성, 검색 코퍼스 확대는 제외한다.

## 가정과 완료 조건

- 컨텍스트 크기는 브라우저와 DB가 보유하는 `user + assistant` 메시지 수다. 현재 API는 매 질문을 독립 검색하므로 과거 턴을 모델에 보내지 않는다.
- 한 정상 턴은 2개 메시지다. 다음 턴으로 400개를 초과할 것이 확실하면 제출 전에 새 conversation ID를 만들고 질문을 그 대화의 첫 턴으로 저장한다.
- 자동 전환 시 `이 대화의 컨텍스트가 가득 차 새 대화에서 계속합니다. 이전 대화는 질문 기록에서 확인할 수 있습니다.`를 접근 가능한 상태 알림으로 표시한다.
- 사이드바 초기 페이지는 최대 20개 대화 요약만 받고, 하단 접근 시 cursor로 다음 페이지를 불러온다. 대화 클릭 전에는 전체 응답 JSON을 받지 않는다.
- 로그인 복원은 동일 Supabase session 조회를 중복하지 않고, 사용자/코퍼스/첫 이력 페이지 중 독립 호출을 병렬화한다.
- 중지 버튼은 브라우저 요청을 AbortController로 취소하고 늦게 도착한 응답을 UI에 반영하지 않는다. 서버/모델 계산의 확정적 중단은 별도 취소 endpoint 없이는 보장하지 않는다.

## 검증 및 롤백

API는 기존 개별 질문 이력 endpoint를 유지해 구 Web을 보호한다. 새 conversation endpoint를 Web만 사용하며 장애 시 기존 endpoint로 되돌릴 수 있다. DB migration은 기존 질문을 각 1턴 대화로 backfill하고 기존 `(user_id, created_at DESC)` 인덱스를 유지한다.

- Production 로그인 지연의 절대 시간은 분산 trace가 없어 코드상 왕복 수만 확정할 수 있다. 변경 후 실제 Vercel/Supabase p50/p95 측정은 운영 관측 작업으로 남을 수 있다.
- HTTP abort가 Vercel Function과 모델 호출까지 전파되는지는 runtime별 검증이 필요하다.
