# 실행 계획 0008: 4단계 검색, 1초 지연 목표, RAG 디버깅

상태: 정확성 구현 완료, Production 지연 최적화 진행 중
작성일: 2026-07-18
소유자: Codex

## 목적과 사용자 결과

자연어 검색을 `모든 핵심어 일치 -> 최소 2개 일치 -> 필수 앵커 + 나머지 핵심어 -> 근거 부족` 순서로 완화한다. 앞 단계에서 충분한 후보를 얻으면 뒤 단계는 실행하지 않고 `skipped_after_success`로 기록한다. 모든 검색 단계가 실패하는 질문은 근거 부족으로 종료한다. 검색 과정과 상위 청크를 재현 가능한 형태로 남기고, AI 생성과 분리한 검색 구간의 Production p95를 1초 이내로 만든다.

## 범위와 비범위

범위는 4단계 키워드 검색, 직접 조문 검색의 선행 처리, 단계별 후보·시간 관측, 고정 실패/성공 질문셋, 원문·청크 수동 검증 절차, PGroonga/PostgreSQL 실행계획과 Supavisor 왕복 비용 측정이다.

LLM 생성 시간, 임베딩 API 시간, 법률 해석 정답 확정, 코퍼스 외 자료 추가는 제외한다. AI 모드에서는 임베딩·생성을 유지하되 이 계획의 1초 목표는 AI 호출 전 검색 구간만 측정한다. 검색 전용 모드에서는 AI를 호출하지 않는다.

## 검색 계약

### 공통 전처리

1. 직접 조문 참조가 있으면 정확한 `문서명 + 조/항/호/목` 경로 검색을 먼저 실행한다.
2. 일반 질문은 NFKC 정규화 후 조사·질문 표현을 제거하고 중복 없는 핵심어를 만든다.
3. 검색 폭발을 막기 위해 점수 상위 핵심어를 최대 5개로 제한한다. 법령명, 정확한 기술기준명, 허가·등록·의무 같은 통제 도메인 용어를 우선한다.
4. 모든 질의는 사용자 문자열을 그대로 실행하지 않고 허용된 토큰으로 다시 구성해 파라미터로 전달한다.

### 1단계: 모든 핵심어 일치

핵심어가 `A, B, C`이면 PGroonga 질의는 `A B C`, 논리식은 `A AND B AND C`다. 정확도 우선 단계이며 후보가 있으면 상위 10건만 반환하고 2·3단계는 실행하지 않는다.

### 2단계: 최소 2개 핵심어 후보 풀

1단계가 0건일 때만 실행한다. 세 핵심어는 `(A AND B) OR (A AND C) OR (B AND C)`로 구성한다. 4~5개 핵심어도 모든 2개 조합을 만들되 최대 10개 조합으로 제한한다. 한 단어만 맞은 문서가 들어오는 전체 OR보다 노이즈를 줄이는 단계다. 이 단계 결과는 바로 반환하지 않고 3단계 앵커 검증의 후보 풀로 사용한다.

핵심어가 하나뿐이면 2단계는 `skipped_insufficient_terms`로 기록한다.

### 3단계: 필수 앵커와 나머지 핵심어

2단계 후보에 항상 적용한다. `최소 2개 일치`가 0건이라면 그보다 좁은 `앵커 AND 나머지`도 새 결과를 만들 수 없으므로 DB를 다시 호출하지 않고 근거 부족으로 간다. 2단계 상위 후보에 앵커가 없을 때만 후보 제한 밖의 누락을 확인하기 위해 앵커 질의를 한 번 실행한다. 앵커 `A`는 다음 우선순위로 하나만 결정한다.

1. 질문에 명시된 정확한 법령·행정규칙·기술기준명
2. 코퍼스의 정확 제목과 일치하는 별칭
3. 통제 어휘에 등록된 도메인 고유어

질의는 `A AND (B OR C)`다. 앵커가 없거나 앵커만 남으면 넓은 단일어 검색을 실행하지 않고 `skipped_no_anchor`로 기록한다. 앵커 선택 근거와 사전 버전을 진단에 저장한다.

### 4단계: 근거 부족

1~3단계가 모두 0건이거나 실행 가능한 3단계 앵커가 없으면 `insufficient_evidence`로 종료한다. 전체 OR 검색은 하지 않는다. 이 상태는 장애가 아니라 검색이 안전하게 답변 생성을 거부한 결과다.

## 1초 지연 목표와 측정 경계

목표는 요청 검증 직후부터 검색 결과가 애플리케이션에 도착할 때까지이며, 질의 정규화·연결 획득·DB 왕복·검색·결과 변환을 포함한다. 인증, Vercel cold start, LLM 임베딩·생성은 별도 구간으로 기록한다.

- 정상 검색과 4단계 근거 부족 모두 Production warm p95 `<= 1,000 ms`
- DB 서버 실행시간 p95 `<= 600 ms`
- 연결 획득 + 모든 네트워크 왕복 p95 `<= 300 ms`
- 정규화 + 결과 변환 p95 `<= 100 ms`
- 동시에 `request_total_ms`, `cold_start_observed`, `retrieval_total_ms`를 기록하여 1초 목표 밖의 지연을 숨기지 않는다.

근거 부족 경로는 세 DB 질의가 실제로 실행되는 최악 경로다. 최소 30회 예열 후 질문별 30회, 전체 300회 이상 측정하고 p50/p95/max를 보고한다. cold 실행은 별도 10회 이상 측정한다. 네트워크 조건과 Vercel/Supabase region, 코퍼스 문서·조문 수를 결과에 함께 남긴다.

## 왕복 비용을 줄이는 순서

1. 먼저 애플리케이션의 1~3단계 순차 호출을 그대로 측정해 단계별 비용을 확인한다.
2. p95가 1초를 넘거나 DB 왕복이 지배적이면 1~3단계를 하나의 PostgreSQL 함수 호출로 옮긴다. 함수 내부에서 단계별 `RETURN QUERY`, 행 수 확인, 조기 반환을 수행해 검색 의미는 유지하면서 애플리케이션-DB 왕복을 최대 3회에서 1회로 줄인다.
3. 단일 호출에도 `stage_used`, `stage_elapsed_ms`, `candidate_count`를 반환하여 관측성을 유지한다.
4. 1·2·3단계를 무조건 `UNION ALL`로 동시에 평가하지 않는다. 성공한 뒤 단계까지 계산하면 순차 완화 계약과 지연 목표를 모두 훼손할 수 있다.

## 인덱스와 SQL 최적화

- 법령 정확 제목 PGroonga 인덱스와 `heading, content` 다중 컬럼 PGroonga 인덱스를 사용한다. 표제/본문 가중치와 인덱스 이름을 명시한다.
- 기준일·현재 버전 제한과 문서/버전/조문 조인에 사용되는 B-tree 인덱스가 실제 필터 순서에 맞는지 확인한다. 추측으로 추가하지 않고 실행계획에서 sequential scan, 예상/실제 행 수 차이를 확인한 뒤 결정한다.
- 각 단계는 점수 계산 전 인덱스 후보를 제한하고 최종 반환은 10건, 디버깅 스냅샷은 상위 5건으로 제한한다. 큰 본문 전체를 진단 JSON에 복제하지 않는다.
- `EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)`으로 실제 실행시간, planning time, index scan, shared hit/read, rows removed를 수집한다. `ANALYZE`는 질의를 실제 실행하므로 읽기 검색 SQL에만 사용한다.
- 통계가 오래되었는지 확인하고 필요한 경우 운영 절차에 따라 `ANALYZE` 후 같은 질문셋을 다시 측정한다. 전역 planner 옵션 변경으로 문제를 감추지 않는다.

## timeout과 운영 관측

- 애플리케이션에는 검색 전체 1,000ms의 절대 deadline을 둔다. 남은 시간이 없으면 다음 완화 단계를 시작하지 않고 `retrieval_deadline_exceeded`로 종료한다.
- DB timeout은 전역 설정 대신 검색 트랜잭션/호출 범위에만 적용한다. 단일 함수화할 경우 `statement_timeout` 후보값은 800ms로 시작하되, Production p95를 측정한 뒤 확정한다.
- `pg_stat_statements`에서 calls, mean/max/total execution time, planning time을 normalized query별로 확인한다. 사용자 질문 원문을 SQL 주석·로그에 넣지 않는다.
- 질문 이력 진단에는 스키마 버전, 정규화/앵커 사전 버전, 코퍼스·파서·청크·임베딩 모델/색인 버전, 단계별 `status/query_shape/elapsed_ms/candidate_count`, 최종 단계와 종료 사유를 남긴다.
- 후보 상위 5건은 `rank, provision_id, document_id, version_id, source_hash, title, provision_path, heading, lexical_score, semantic_score, fused_score`만 저장한다. 본문은 해당 고정 version/provision을 조회해 검토하여 개인정보·원문 중복 로그를 피한다.

## 연결 풀과 prepared statement 제약

Vercel의 임시 연결은 Supavisor transaction mode를 사용한다. 서버리스 트래픽에 맞고 연결 재사용으로 생성 비용을 줄이지만 transaction mode는 prepared statements를 지원하지 않으므로 SQLAlchemy/asyncpg statement cache를 0으로 유지하고 named prepare를 성능 해법으로 사용하지 않는다. 파라미터 바인딩은 계속 사용한다.

Vercel Function region은 Supabase DB와 같은 지역 또는 가장 가까운 지역으로 고정한다. p95의 연결 획득 시간이 높으면 Supavisor pool 대기와 실제 사용량을 먼저 확인한다. pool 크기는 다른 Supabase 서비스의 연결 여유를 포함해 관측 후 조정하며, 단지 지연을 낮추려고 임의로 키우지 않는다. 장기 상주 backend로 구조가 바뀔 때만 direct/session 연결과 prepared statement의 planning 절감 효과를 별도 재평가한다.

## 고정 디버깅 질문셋

Production에서 실패했던 질문과 성공/경계 사례 8개를 v1 평가셋으로 고정한다.

1. `전기저장시설 설치 시 확인할 기준은?`
2. `에너지 사업 허가 절차를 알려주세요`
3. `전기사업법 제1조 2항에 대해 알려주세요`
4. `흐음`
5. `전기사업법 제7조 제1항은 어떤 허가를 규정하나요?`
6. `전기사업 허가를 신청할 때 제출해야 하는 서류는 무엇인가요?`
7. `ESS 화재안전 기술기준은 무엇인가요?`
8. `분산에너지 사업자의 등록 요건은 무엇인가요?`

3·4번은 직접 조문 경로 부재와 모호한 단일어의 근거 부족을 측정한다. 평가셋에는 기대 문서/조문 또는 `no_evidence`, 정규화 핵심어, 앵커, 파이프라인 버전을 명시한다.

## RAG 디버깅 보고서 계약

검색과 생성을 분리하고 질문별로 다음 데이터를 빠짐없이 보고한다.

### 1. 실패 상황과 버전 고정

- 원본 질문, 요청 시각, 계정 내부 ID 또는 비식별 평가 ID
- API 배포 commit, 검색 계약/정규화/앵커 사전 버전
- 코퍼스 기준시점과 문서·버전·조문 수
- parser/chunk schema, PGroonga/pgvector index, 임베딩 모델·차원 버전
- 요청 모드, AI 활성 상태, LLM 모델과 생성 결과/실패 사유

### 2. Retrieve 검증

- 기대 답을 뒷받침하는 원문이 고정 source/version에 존재하는지: `yes/no/unknown`과 provision ID
- 실제 검색된 상위 1~5 청크의 ID, 제목, 경로, 표제, 점수와 원문 직접 검토 결과
- 청크별 `relevant/partial/noise`, 문맥 단절·과대 청크·잘못된 구조 경로 여부
- 기대 핵심 문서의 실제 순위, 10위 밖 여부, 후보 자체 누락 여부
- 누락 지점을 `수집/파싱/버전 활성화/인덱스/필터/순위` 중 하나로 분류

### 3. Query 변환과 하이브리드 검증

- 원문 -> 정규화 토큰 -> 제거 토큰과 이유 -> 선택 앵커와 근거
- 단계별 실행 query shape, 실제 실행/skip 이유, 후보 수, 시간
- AI 모드이면 lexical 상위 5, semantic 상위 5, RRF 상위 5와 순위 변화
- 검색 전용이면 semantic을 `not_attempted_search_only`로 명시
- 벡터가 0건이면 임베딩 미생성, 차원/모델 불일치, 임계값, 필터 중 원인을 분리

### 4. 생성과 검색 원인 분리

- 검색 상위 청크에 충분한 근거가 있었는지 먼저 판정
- 근거가 있었는데 답이 실패하면 generation/structured output/citation gate 원인으로 분류
- 근거가 없으면 모델 프롬프트 조정으로 우회하지 않고 retrieval/chunk/corpus 조치로 연결
- 질문별 최종 원인, 재현 증거, 수정 후보, 수정 후 같은 고정 질문 회귀 결과를 기록

## 검증 및 롤백

새 단계형 검색은 feature flag로 기존 검색과 비교하되 사용자 응답에는 한 결과만 사용한다. Recall@5 저하, 근거 부족 오탐 증가, warm p95 1초 초과 시 flag를 되돌린다. 인덱스 변경은 먼저 Production과 같은 데이터의 staging에서 실행계획을 확인하고, 기존 인덱스는 새 인덱스가 검증될 때까지 삭제하지 않는다.

- 2026-07-18: 4단계는 모두 상태를 기록하되 성공 뒤 단계는 실행하지 않는다. 실제 SQL을 모두 실행하는 것은 정확도 향상이 없고 지연만 늘린다.
- 2026-07-18: 초기 구현은 이해하기 쉬운 순차 호출로 측정하고, 왕복이 병목으로 확인될 때만 하나의 DB 함수로 합친다.
- 2026-07-18: prepared statement는 Supavisor transaction mode 제약 때문에 최적화 후보에서 제외한다.

## 진행 기록

- 2026-07-18: 공식 PGroonga 문서에서 공백 AND, 명시적 OR, 다중 컬럼 가중 검색과 인덱스 계약을 확인했다.
- 2026-07-18: PostgreSQL 공식 문서에서 `EXPLAIN ANALYZE`의 실제 실행, `statement_timeout`, generic/custom prepared plan 제약을 확인했다.
- 2026-07-18: Supabase 공식 문서에서 서버리스용 transaction pooler, prepared statement 미지원, `pg_stat_statements` 관측과 pool 설정 주의점을 확인했다.
- 2026-07-18: Production `0004` 적용과 재수집을 완료했다. 9문서·9버전·3,066조항, 임베딩 0건, 평가 실행 0건이다.
- 2026-07-18: 최종 고정 질문 8개는 기대 계약을 모두 통과했다. 검색 시간은 min 90.980ms, p50 1,031.311ms, p95/max 5,462.805ms였고 최악 질문의 최소 2개 조합 단계가 4,296.774ms를 사용했다.
- 2026-07-18: 임의 최장어 앵커와 동치 질의 재호출을 제거했다. 근거 부족 단일어 질문 10회는 min 864.666ms, p50 872.739ms, p95/max 942.486ms로 10/10회 1초 이내였다. 이 경로는 최초 검색 한 번만 DB에서 실행하고 나머지는 `skipped_duplicate_query`, `skipped_no_anchor`로 기록한다.

- Vercel과 Supabase의 실제 region 및 Production connection mode를 배포 설정에서 확인해야 한다.
