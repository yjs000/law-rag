# 실행 계획 0005: 로그인·익명 사용자 전체 흐름 엣지케이스

## 목적과 사용자 결과

실제 개인 계정 대신 가짜 사용자 ID와 테스트 인증 경계를 사용해 로그인 사용자와 익명 사용자가 첫 화면부터 질문·결과·이력·로그아웃까지 안전하게 완료하도록 한다. 각 흐름의 정상·실패·경계 사례를 자동화하고 발견된 결함을 보완한다.

## 범위와 비범위

범위는 Web 인증 상태 전이, OAuth callback 입력 검증, 익명 질문 비저장, 로그인 질문 이력과 소유권, 로그아웃 UI 초기화, quota 경계, 오류·빈 결과·재시도 테스트와 관련 문서 갱신이다.

실제 Google 계정 로그인, 운영 계정 삭제, 운영 데이터 변경, 두 실제 사용자를 이용한 검증은 범위에서 제외한다. 모든 인증 테스트는 가짜 ID·토큰·IP를 사용한다.

## 측정 가능한 완료 조건

- 익명 사용자는 질문·빈 결과·오류 후에도 계속 질문할 수 있고 질문 이력이 생성되지 않는다.
- 로그인 사용자는 자신의 질문만 저장·조회·삭제하며 다른 가짜 사용자 ID의 이력에 접근할 수 없다.
- 로그인·로그아웃·세션 만료·다중 탭 상태 변경 때 UI와 민감한 현재 대화가 인증 상태와 일치한다.
- OAuth callback의 외부 redirect, host 변조, code 누락과 교환 실패가 안전하게 차단된다.
- 정상·실패·경계 테스트, lint, typecheck, production build와 저장소 전체 검증이 통과한다.
- 변경 문서와 기술 브리핑이 실제 구현·검증 결과와 일치하며 브랜치가 원격에 push된다.

## 검증 및 롤백

대표 명령은 `scripts/verify.ps1`이다. Web은 lint·typecheck·Vitest·production build를, API는 pytest를 실행한다. 외부 인증 설정이나 운영 데이터는 변경하지 않는다. 실패 시 이 브랜치의 변경만 되돌리며 익명 질문 경로를 유지한다.

## 미결정과 차단 요소

- 실제 두 계정의 Supabase RLS와 운영 계정 삭제는 파괴적·다중 사용자 검증이므로 이번 계획에서는 가짜 ID 기반 통합 테스트로 대체한다.
- 외부 OAuth 공급자의 CAPTCHA·2단계 인증·사용자 동의 화면 자체는 자동화하지 않는다.
