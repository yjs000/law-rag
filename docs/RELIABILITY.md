# 신뢰성

## 사용자 관점의 핵심 기능

1. 질문을 접수한다.
2. 현재 색인에서 근거를 검색한다.
3. 근거에 연결된 답변을 반환한다.
4. 인용 원문을 연다.

## 초기 SLI/SLO 제안

실제 트래픽 측정 전 목표값은 승인되지 않은 초안이다.

| 사용자 여정 | SLI | 초기 제안 |
|---|---|---|
| 검색 | 성공 응답 비율 | 월 99.9% |
| 답변 | 검증된 답변 또는 명시적 근거 부족 반환 | 월 99.5% |
| 인용 열기 | 유효 원문 위치 반환 | 월 99.9% |
| 최신성 | 출처 게시부터 검색 가능까지 지연 | 출처별 별도 정의 |
| 성능 | 첫 유의미 결과 p95 | 제품 요구 확인 후 결정 |

정확하지 않은 답변은 HTTP 성공이어도 신뢰성 성공으로 계산하지 않는다.

## 성능 저하 전략

- 생성 모델 장애: 검색 결과와 원문 링크 제공
- 재순위 장애: 기본 하이브리드 순위 사용 및 표시
- 벡터 검색 장애: 키워드 검색으로 제한적 동작
- 검색 전체 장애: 생성 답변 금지, 재시도 가능한 오류
- 최신성 초과: 마지막 성공 기준 시점 경고

## 관측 가능성

요청 ID와 추적 ID로 API, 검색, 모델 호출, 인용 검증을 연결한다. 로그는 구조화하고 원문 질문, 법률 원문 전문, 개인 식별정보는 기본 제외한다. 핵심 메트릭에는 단계별 지연, 후보 수, 결과 없음, 모델/검증 오류, 색인 최신성, 질의당 비용을 포함한다.

## 운영 준비 체크

- 의존 서비스별 시간 제한, 제한된 재시도, 회로 차단
- 단일 클라우드 서버 안에서 Web/API/collector를 독립 프로세스로 실행하고 collector 실패가 질문 API를 재시작하지 않도록 격리
- 단일 서버 장애가 전체 서비스 장애가 되는 위험을 모니터링하고 후속 다중화 필요성을 운영 지표로 판단
- 멱등적인 수집·색인 작업과 체크포인트
- 색인 버전 원자적 승격 및 이전 버전 롤백
- DB 백업/복구 시험과 목표 RPO/RTO
- 장애 등급, 연락, 사용자 공지, 사후 분석 템플릿
