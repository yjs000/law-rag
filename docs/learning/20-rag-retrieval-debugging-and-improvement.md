# 20 RAG 검색 시스템 디버깅과 로직 개선

## 목적

이 문서는 운영 질문이 모두 답변 실패한 상황에서 검색과 생성을 분리해 원인을 찾은 과정, 실제로 변경한 검색 로직, 검증 결과와 아직 끝나지 않은 성능 작업을 재현 가능하게 기록한다. 법률 답변 내용은 모델 기억으로 보완하지 않고 국가법령정보 공동활용 Open API에서 수집한 고정 코퍼스만 사용한다.

## 디버깅 원칙

RAG 실패를 한 덩어리의 `답변 실패`로 보지 않고 다음 경계로 분리한다.

1. 입력: 질문 원문, 요청 모드, 기준일, 인증 주체를 확인한다.
2. 파싱: 직접 조문 참조와 자연어 질문을 구분하고 파싱 결과를 확인한다.
3. 질의 변환: 정규화된 핵심어, 제거된 질문 표현, 별칭, 앵커와 실제 PGroonga 질의를 확인한다.
4. 코퍼스: 기대 원문·버전·조항이 DB에 존재하고 기준일 필터를 통과하는지 확인한다.
5. 검색: 단계별 원시 후보 수, 애플리케이션 재검증 후 후보 수, 상위 1~5 청크와 시간을 확인한다.
6. 의미 검색: AI 모드에서만 임베딩 생성 여부, 벡터 후보, lexical/semantic 병합을 확인한다. 검색 전용 모드는 `not_attempted_search_only`로 구분한다.
7. 생성: 검색 근거가 충분한데도 실패한 경우에만 LLM 생성, 구조화 출력과 인용 게이트를 조사한다.
8. 결과: 최종 상태를 근거 반환, 근거 부족, 검색 장애, 생성 장애, 인용 검증 실패로 분류한다.

검색 후보가 없는데 프롬프트나 모델을 바꾸는 방식으로 우회하지 않는다. 먼저 원문 누락, 청킹, 질의 변환, 필터, 인덱스와 순위를 고친다.

## 최초 운영 상태와 실패 재현

운영 질문 이력에서 자연어 질문들이 근거를 한 건도 반환하지 못했다. 당시 Production 상태는 다음과 같았다.

- 법률 문서 9개와 조항 3,050개가 존재했다.
- 임베딩은 0건이고 AI는 `ai_disabled`였다.
- 질문 전체를 PGroonga에 전달해 공백으로 분리된 모든 표현이 일치해야 했다.
- 제목과 조문 표제가 검색 대상에서 충분히 활용되지 않았다.
- `확인할`, `알려주세요`, `절차`처럼 원문에 없을 가능성이 큰 질문 표현까지 필수어가 됐다.
- 검색·파싱·생성 단계별 기록이 없어 최종 0건만으로는 실패 위치를 구분할 수 없었다.

재현 결과 `전기저장시설 설치 기준`은 후보가 있었지만 `확인`을 추가하면 0건이 됐고, `에너지 사업 허가`는 후보가 있었지만 `절차`를 추가하면 0건이 됐다. AI 비활성 자체가 키워드 검색 0건의 직접 원인은 아니었고, 검색 전용 경로의 질의 변환과 검색 범위가 구조적 원인이었다.

## 발견한 문제와 개선

### 1. AI 비활성과 검색 전용 모드가 섞여 있었다

검색 전용 모드가 AI 상태에 의존하면 임베딩이나 생성 모델을 연결하지 않은 동안 모든 질문이 실패할 수 있다.

개선 후 `/v1/search`와 `search_only`는 임베딩·생성 모델을 호출하지 않고 PGroonga 키워드 검색만으로 동작한다. Terra 모드에서 AI가 활성화된 경우에는 임베딩과 생성 호출을 유지한다. AI 장애는 검색 장애와 다른 `fallback_reason`으로 기록한다.

### 2. 자연어 전체가 엄격 AND 질의가 됐다

질문을 NFKC로 정규화하고 한글·영문·숫자 토큰만 허용한다. 조사, 질문형 어미와 `알려`, `무엇`, `설명`, `확인`, `절차` 같은 검색 의도 표현을 제거한다. 사용자 문자열을 PGroonga 문법으로 직접 실행하지 않고 서버가 허용 토큰과 별칭으로 질의를 다시 만든다. 핵심어는 조합 폭발을 막기 위해 최대 5개로 제한한다.

### 3. 제목과 표제가 검색 근거에서 빠졌다

Migration `0004`에서 법령 제목, 조문 표제와 본문을 함께 검색하도록 PGroonga 함수와 인덱스를 갱신했다. 질문 이력에는 검색 전략과 단계별 후보 수를 저장하도록 진단 JSONB를 추가했다.

### 4. 전체 OR 완화는 너무 많은 노이즈를 허용했다

처음에는 엄격 AND가 0건이면 모든 핵심어를 OR로 검색했다. 이 방식은 일반어 하나만 일치한 문서도 근거가 되어 법률 답변의 정밀도를 낮춘다. 전체 OR를 제거하고 다음 4단계로 변경했다.

1. 직접 조문 경로: `법령명 + 제N조/항/호/목`이 있으면 자연어 검색보다 먼저 정확 경로를 조회한다.
2. 모든 핵심어: `A AND B AND C`가 모두 확인된 후보만 반환한다.
3. 최소 2개 후보 풀: `(A AND B) OR (A AND C) OR (B AND C)`로 후보를 모으되 바로 반환하지 않는다.
4. 앵커 필수: 2개 이상 일치 후보에 정확 법령·기술기준명, 카탈로그 제목 별칭 또는 통제 도메인 용어가 있는지 확인한다. 없으면 근거 부족이다.
5. 근거 부족: 안전한 후보가 없으면 전체 OR로 더 넓히지 않고 `insufficient_evidence`로 종료한다.

여기서 최소 2개 후보가 0건이면 더 좁은 `A AND (B OR C)`도 새 결과를 만들 수 없다. 따라서 3단계는 2단계의 품질 게이트로 사용하고, 상위 후보 제한 때문에 앵커 후보가 잘렸을 가능성이 있을 때만 앵커 질의를 추가 실행한다.

### 5. 임의 최장어가 앵커가 됐다

카탈로그와 일치하지 않는 질문에서도 가장 긴 단어를 앵커로 강제해 유효한 후보를 버릴 수 있었다. 앵커를 카탈로그 제목과 실제 통제 도메인 어휘로 제한했다. `흐음`처럼 앵커가 없는 질문은 `skipped_no_anchor` 후 근거 부족으로 끝난다.

### 6. 논리적으로 같은 질의를 반복 실행했다

한두 개 핵심어 질문에서는 모든 핵심어, 최소 2개, 앵커 질의가 문자열만 다르고 논리적으로 같을 수 있었다. 실행한 질의 집합을 추적해 동치 질의를 `skipped_duplicate_query`로 처리하고 실제 마지막 실행 질의만 진단에 기록한다. `흐음` 근거 부족 경로는 DB 검색 한 번만 수행한다.

### 7. 직접 조문과 기술기준 번호를 같은 구조로 볼 수 있었다

`제11조`는 법령의 조문 경로이고 `1.1`은 NFTC/NFPC 기술기준 내부 절 번호다. 직접 조문 파서는 `제N조`, `제N항`, `제N호`, `제N목`을 구조 경로로 처리한다. 기술기준 파서는 `1.1`, `1.2` 같은 절을 별도 청크 경계로 사용한다.

### 8. 기술기준 본문 인용을 새 청크로 오인했다

NFTC 원문이 큰 문자열 하나로 들어와 처음에는 과대 청크가 됐다. 절 번호 분할을 추가한 뒤에는 본문의 `2.2에도`, `2.2 및`, `2.3을` 같은 인용까지 새 표제로 오인해 중복 경로가 발생했다. 조사와 접속 표현이 뒤따르는 번호를 표제에서 제외하도록 파서를 수정했다. Production 재수집 후 9개 문서, 9개 버전, 3,066개 조항을 확인했고 NFTC 16개, NFPC 15개 기술기준 절로 분리됐다.

### 9. 실패 위치를 관측할 수 없었다

질문 진단을 `input_validation`, `parsing`, `embedding`, `retrieval`, `generation`, `outcome`으로 나눴다. 검색 trace에는 다음을 기록한다.

- 정규화 질의와 핵심어
- 선택된 앵커
- 실행 전략과 실제 마지막 실행 질의
- 단계별 query, status, 원시 후보 수, 승인 후보 수, 시간
- 전체 검색 시간과 최종 후보 수
- 직접 조문 제목·경로

인증정보, API 키, 오류 전문과 원문 전문은 로그에 추가하지 않는다. 상위 청크 원문을 검토할 때는 고정된 provision/version을 별도로 조회한다.

### 10. 빈 직접 조문 검색의 안내가 실제 검색 범위를 충분히 설명하지 않았다

법령명 없는 `1조 2항`도 기준일에 유효한 MVP 대상 법령 전체에서 같은 경로를 조회한다. 따라서 0건 안내가 법령명을 추가하라고만 하면 법령명 누락 때문에 검색을 실행하지 못한 것으로 오해할 수 있었다. 법령명 없는 요청은 전체 대상 법령을 검색했다는 사실을 표시하고, 법령명이 있는 요청은 해당 법령에서 경로를 찾지 못했다고 구분했다. 이 안내는 생성 모델 없이 결정된다.

사용자 입력의 원형 숫자 항 표기(`제1조 ②항`, `제1조제②항`)도 숫자 경로와 Open API 기호 경로로 함께 정규화했다. 이전에는 항 표기를 버리고 `제1조`만 검색할 수 있어 잘못된 상위 경로 결과를 반환할 위험이 있었다.

### 11. 복수·범위 조문은 단일 경로 파서의 반복 호출로 안전하게 처리할 수 없었다

`제일조 제이항`, `제1조 제2항 및 제3항`, `제1조부터 제3조`를 각각 한글 수사 정규화, 공유 조 번호를 가진 복수 항, 순서가 보존된 경로 집합으로 파싱한다. 저장소는 집합에 포함된 정확 경로만 조회하며 PostgreSQL은 JSON 경로 배열의 `WITH ORDINALITY` 순서로 결과를 정렬한다.

범위 폭은 20개 조로 제한했다. `제3조부터 제1조` 같은 역방향이나 `제1조부터 제21조` 같은 초과 범위는 직접 조문 질의로 인식하되 결과 집합을 비워 둔다. 이를 일반 자연어 검색으로 돌리거나 앞 20개만 반환하면 사용자가 요청하지 않은 근거 또는 불완전한 범위를 정확한 결과처럼 보여줄 수 있기 때문이다.

## 고정 평가셋과 결과

운영 실패, 직접 조문 성공, 자연어 성공과 근거 부족을 포함한 8개 질문을 `apps/api/evaluation/retrieval-debug-v1.json`에 고정했다.

| 질문 유형 | 기대 결과 | 최종 결과 | 검색 시간 |
|---|---|---|---:|
| 전기저장시설 설치 기준 | NFPC/NFTC 근거 | 통과 | 1,211ms |
| 에너지 사업 허가 절차 | 전기사업 관련 법령 | 통과 | 1,031ms |
| 존재하지 않는 제1조 2항 | 근거 부족 | 통과 | 93ms |
| 모호한 단일어 `흐음` | 근거 부족 | 통과 | 870ms |
| 전기사업법 제7조 제1항 | 정확 조문 | 통과 | 91ms |
| 허가 신청 제출 서류 | 시행규칙 | 통과 | 1,201ms |
| ESS 화재안전 기술기준 | NFTC | 통과 | 1,839ms |
| 분산에너지 등록 요건 | 특별법 근거 | 통과 | 5,463ms |

8개 질문의 기대 계약은 모두 통과했다. 전체 검색 시간은 min 90.980ms, p50 1,031.311ms, p95/max 5,462.805ms였다. 최악 질문은 2단계 최소 2개 조합 검색에 4,296.774ms를 사용했다.

근거 부족 단일어를 10회 반복했을 때 min 864.666ms, p50 872.739ms, p95/max 942.486ms였고 10회 모두 1초 이내였다. 이 결과는 단순 근거 부족 경로의 중복 왕복은 제거됐지만 복합 조합 검색의 DB 실행시간은 해결되지 않았음을 뜻한다.

## 완료하지 못한 성능 작업과 이유

### EXPLAIN과 인덱스 검증

실행하지 않았다. 단계별 애플리케이션 시간만 측정하고 DB 내부 planning/execution, buffer hit/read와 scan 종류를 분리하지 않은 채 구현·검증·배포를 먼저 마쳤다. Production 읽기 질의에 한정하면 실행 가능한 작업이었으므로 외부 차단이 아니라 작업 순서와 완료 판단의 문제다.

### 단일 DB 함수와 재측정

구현하지 않았다. 계획은 p95가 1초를 넘으면 진행하는 조건이었고 전체 평가 p95가 5.46초이므로 조건은 이미 충족됐다. 다만 EXPLAIN 없이 왕복만 원인이라고 가정해 Production 함수를 추가하면 4.30초인 조합 질의 자체를 해결하지 못한 채 복잡도만 늘릴 수 있다. 먼저 DB 실행시간과 네트워크 왕복을 분리한 뒤 함수화 여부를 결정해야 한다. 이 선행관계는 타당하지만, 미완료 상태를 더 분명히 보고하지 않고 커밋을 종료한 것은 잘못이다.

### Vercel/Supabase region과 Supavisor pool 대기

확인하지 않았다. 저장소에는 transaction pooler와 statement cache 0 계약이 있지만 실제 Vercel 프로젝트 region, Supabase region, pool mode와 대기 지표는 배포 플랫폼 설정과 운영 대시보드 확인이 필요하다. 당시 Production DB 접근 승인은 데이터 migration·재수집·읽기 평가에 사용했고 배포 설정 확인까지 이어가지 않았다.

### 회귀·전체 검증·결과 기록

실제로 완료했지만 체크박스를 갱신하지 않았다. API 148개, 코어 3개, 수집기 34개, Web 35개 테스트와 Ruff, ESLint, TypeScript, 문서 검사, Production build가 통과했고 실제 측정값도 실행 계획에 기록했다. 이는 구현 누락이 아니라 문서 드리프트다.

## 남은 실행 순서와 완료 조건

1. 최악 질문의 1단계와 2단계에 `EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)`을 실행한다. PGroonga/B-tree index scan, planning/execution time, buffer hit/read, rows removed를 저장한다.
2. 애플리케이션 전체 시간에서 DB 실행시간을 빼 연결 획득과 네트워크 왕복을 계산한다. `pg_stat_statements`의 calls, mean/max/total execution time도 함께 확인한다.
3. Supabase는 서울 `ap-northeast-2`, runtime transaction mode 6543, migration session mode 5432까지 확인됐다. Vercel Function region과 Supavisor pool 대기를 운영 설정에서 추가 확인한다.
4. 왕복이 지배적이면 1~3단계를 조기 반환하는 PostgreSQL 함수 한 번으로 합친다. 2단계 SQL 자체가 지배적이면 먼저 query shape, 인덱스와 후보 제한을 최적화한다.
5. 변경 후 고정 8개 질문과 warm 300회 이상, cold 10회 이상을 재측정한다. 정상 검색과 근거 부족 검색의 Production warm p95가 모두 1초 이내여야 완료다.
6. 전체 테스트, lint, typecheck, 문서 검사와 Production build를 다시 실행하고 결과를 실행 계획에 기록한다.

## 재현 명령

`apps/api`에서 실행한다. Production DB 명령은 읽기 전용이라도 운영 접근 승인이 필요하다.

```text
cmd /d /c ..\..\.venv\Scripts\python.exe -m scripts.debug_retrieval_pipeline --output ..\..\.data\retrieval-debug-after-0004.json
cmd /d /c ..\..\.venv\Scripts\python.exe -m scripts.benchmark_retrieval_latency --case-id ambiguous-filler --repetitions 10
cmd /d /c ..\..\.venv\Scripts\python.exe -m pytest -q
```

상세 성능 실행 계획은 [실행 계획 0008](../exec-plans/active/0008-four-stage-retrieval-latency-and-debugging.md), 현재 검색 계약은 [RAG 파이프라인 설계](../design-docs/rag-pipeline.md)를 따른다.

현재 측정은 8개 질문 각 1회와 근거 부족 질문 10회의 소표본 진단이다. 계획한 30회 예열, 질문별 30회, 전체 300회와 cold 10회는 아직 수행하지 않았으므로 Production p95 목표 달성으로 일반화하지 않는다.
