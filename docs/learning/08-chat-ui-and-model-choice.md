# 08 채팅 UI와 모델 선택 경계

작성일: 2026-07-14

## 무엇을 바꿨나

법령 질문·답변·원문을 서로 다른 고정 열에 두던 화면을 하나의 채팅 흐름으로 옮겼다. 데스크톱의 왼쪽 목록은 로그인 사용자의 질문 이력을 열고, 모바일에서는 닫힌 사이드바로 바뀐다. 질문 작성기는 화면 아래에 유지되며 원문 종류와 기준일을 함께 전달한다.

## 왜 요청 모드가 서버 계약이어야 하나

화면에서 `검색 전용`을 선택해도 서버가 기존처럼 AI 가용성만 보고 모델을 호출한다면 선택지는 장식에 불과하다. 그래서 `QuestionRequest.answer_mode`를 `terra | search_only`로 제한하고, 검색 전용이면 임베딩·생성 답변 경로를 건너뛴다. 기본값은 기존 클라이언트 호환을 위해 `terra`다.

`other-model` 같은 값은 시스템 경계의 Pydantic 검증에서 422로 거절한다. UI의 다른 모델 항목도 비활성으로 표시한다. 이는 모델을 추가하려면 먼저 품질 평가와 인용 게이트 검증이 필요하다는 뜻이다.

## 익명과 로그인 상태

익명 질문은 같은 채팅 UI에서 정상 답변되지만 서버 질문 이력에 저장되지 않는다. 첫 답변 뒤 로그인 안내도 현재 답변을 덮지 않는 인라인 카드다. 로그인 전 질문을 로그인 뒤 소급 저장하지 않는 이유는 사용자 동의와 계정 연결 시점을 명확히 하기 위해서다.

Google 첫 로그인은 가입과 로그인을 한 단계로 처리한다. 계정 대시보드는 확정된 정보만 표시한다. 계정별 운영 한도가 아직 정해지지 않았으므로 임의 숫자를 만들지 않고 `미결정`으로 둔다.

## 직접 검증

```powershell
pnpm lint:web
pnpm typecheck
pnpm test:web
pnpm build:web
uv run --project apps/api pytest apps/api/tests
```

검색 전용 API 테스트는 생성 답변 어댑터가 한 번도 만들어지지 않는지 확인한다. 경계 테스트는 미지원 모델 값이 422인지 확인한다. 프론트 검증은 API 직렬화, 안전한 텍스트 렌더링, 이력·내보내기 흐름을 유지한다.

## 전송 메시지와 다음 초안 분리

작성기 문자열을 그대로 전송 상태로 사용하면 말풍선이 생긴 뒤에도 같은 문장이 입력창에 남는다. 제출 시 정규화한 문장은 `submittedQuestion`에 고정하고 작성기 `question`은 즉시 빈 문자열로 바꾼 뒤 입력 포커스를 돌려준다. 네트워크 응답은 작성기 상태를 갱신하지 않으므로, 사용자는 기다리는 동안 다음 질문을 작성해도 초안을 잃지 않는다. 현재 화면은 응답 상태가 하나이므로 처리 중에는 추가 전송만 막고 텍스트 입력은 계속 허용한다.
